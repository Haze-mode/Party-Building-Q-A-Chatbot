基于GLM-4的党建知识智能问答系统

毕业设计项目 · 模型微调 + 知识库检索增强 + Web全栈实现

项目概述

本项目设计并实现了一个面向党建领域的智能问答系统。核心思路是采用模型微调 + 知识库检索增强（RAG）双重机制：通过LoRA微调让GLM-4-9B-Chat掌握党建专业术语和规范表述风格，通过知识库检索为模型提供最新政策文件原文作为回答依据，两者互补，兼顾专业性与时效性。

系统采用前后端分离架构，后端基于Tornado异步框架，前端为原生HTML/CSS/JavaScript实现，包含用户认证、多轮对话、知识库管理、会话管理和缓存优化等完整功能模块。

核心模块实现

1. 模型微调（LoRA）

  • 基座模型：GLM-4-9B-Chat
  • 微调方法：LoRA，秩 r=16，仅训练 0.11% 参数（约1081万 / 94亿）
  • 训练数据：数百条党建领域对话样本，覆盖党章党规、党史、政策解读、党务工作
  • 数据格式：system/user/assistant 角色化多轮对话，system 设为"党建知识专家"
  • 训练配置：800步，有效批次8，学习率5e-5，warmup+cosine衰减，梯度检查点+梯度累积
  • 评估方式：Jieba分词后的ROUGE-1/2/L和BLEU-4自动评估 + 人工定性对比
  • 最终模型：checkpoint-600（验证集ROUGE-1=4.42, ROUGE-L=1.34）

2. 知识库检索

  • 支持格式：PDF、Word (DOCX)、TXT
  • 文档解析：PyPDFLoader + python-docx + 直接读取
  • 分块策略：段落优先切分，超长段落按512字符递归切分，相邻块重叠50字符
  • 检索算法：scikit-learn TfidfVectorizer 构建词项-文档权重矩阵
  • 相似度计算：查询向量与文档向量的点积，返Top-K（默认K=3）
  • 索引更新：管理员触发全量重载（重新解析所有文档并重建索引）
  • 架构预留：代码中已实现FAISS向量检索逻辑，换用带完整编译链的环境可直接启用

3. 缓存系统

  • 两个独立LRU缓存：检索缓存（容量1000）+ 回答缓存（容量500）
  • 实现：OrderedDict + MD5哈希键 + TTL过期（默认3600秒）
  • 后台定时清理过期条目，知识库更新时主动清空全量缓存
  • 效果：缓存命中后响应时间从6-10秒降至0.2秒以内

4. 会话管理

  • 以session_id为键在内存中维护对话状态，与用户认证解耦
  • 支持游客模式（guest_前缀），登录用户以用户名为session_id
  • 多轮对话：保留最近10轮历史，按"系统消息→历史→当前问题"拼接
  • TTL=3600秒，后台每10分钟清理过期会话

5. Web服务与前端

  • 后端：Tornado异步框架，监听6006端口，提供RESTful API
  • 认证：UUID Token + SHA256密码哈希，游客/普通用户/管理员三级权限
  • 前端：三个页面（聊天、登录、知识库管理），党建红色主题，响应式布局
  • 功能：多轮对话、会话历史查看、导出DOC、流式输出（SSE）
  • Mock模式：无需GPU即可跑通前后端联调，返回预设模拟回答

已知局限与改进方向

  1. 训练数据规模有限：数百条样本训练800步（约22个epoch）导致过拟合，验证集指标在600步后骤降。后续需扩充数据、引入早停机制。
  2. FAISS未成功部署：因运行环境缺少编译依赖，当前线上使用TF-IDF关键词检索。代码中已实现FAISS检索逻辑，换环境可启用。
  3. TF-IDF语义理解局限：基于词面匹配，对同义词和复杂长查询效果有限，后续可升级为Embedding稠密向量检索（如BGE + FAISS）。
  4. 自动评估指标不理想：ROUGE/BLEU在党建领域区分度低（BLEU-4最高仅0.0013），当前主要依赖人工定性评估，后续需引入BERTScore等语义相似度指标。
  5. 知识库覆盖范围有限：当前仅包含基础党建文献，需持续扩充。
  6. 容器化未完成：项目在AutoDL上直接运行，未编写Dockerfile。
  7. 缓存细粒度失效未实现：当前采用全量清空策略，可优化为仅清除受影响条目。
