## 任务二：多余文件清理建议

**原则：文字类文件可以大胆整理、合并、删除；代码类文件务必保守，只标注“确定没问题可以删”的项，不确定的宁可不列。**

- 项目根目录 README.md 保留；若存在 README_en.md 或类似英文版、旧版说明，直接建议删除。

### 代码类文件——只标注“确定能删”的，执行严格判断标准

以下情况可以“确定能删”：
1. **文件名明确表明是临时/缓存/备份**：如 `*_backup.py`、`*.bak`、`__pycache__/` 等。
2. **明确的重复依赖文件**：如同目录下同时存在 `requirements.txt` 和 `requirements1.txt` 且内容高度重复，删除后者。
3. **功能已被确认替代且不再被引用的文件**：
   - 你实际运行的微调脚本是 `finetune1.py`，若同一目录下还有 `finetune.py`，且经核查 `finetune1.py` 不 import `finetune.py`，则可删 `finetune.py`。
   - 你实际用 `lora1.yaml`，若目录下还有 `lora2.yaml`、`sft.yaml`、`ptuning_v2.yaml` 等未使用的配置文件，且无其他脚本引用它们，则可删。
   - 你实际后端是 `web_server.py`，若存在旧的独立服务脚本（如 `chat-robot-glm4-9b.py`），且不再被 import 或调用，则可删。
   - `knowledge_base.py` 和 `knowledge_base_faiss.py` 功能重叠，论文采用 TF-IDF 方案，若经核查 `knowledge_base_faiss.py` 未被其他地方 import，则可删。

以下情况 **不要删除**，只可标注“建议归档”或宁可不列：
- 不确定是否被其他文件 import 的脚本。
- 多个 checkpoint 目录（如 checkpoint-400/600/800），即使论文只用 600，也不建议直接删，可标注“建议归档”。
- 数据构建、评估脚本等可能有复现价值的，只归档，不建议删。
